# fix-issue

## 目标
在不扩大改动面的前提下修复一个明确问题，并用测试或运行态验证确认修复生效。

## 适用场景
- 用户明确要求“修这个问题”
- issue 已定位到具体模块或链路
- 需要尽量少改代码，同时保证可回归

## 修复原则
- 先定位根因，再改代码
- 优先最小修复，不顺手重构
- 修复必须配验证，不能只改不测
- 与 `AGENTS.md` 的稳定性、兼容性和文档同步要求保持一致

## 推荐工作流
1. 复现问题，确认失败点
2. 定位根因，区分直接原因和深层原因
3. 设计最小修复方案
4. 先补测试，再做代码改动，保持 TDD
5. 运行相关测试和必要的 py_compile / CI gate
6. 如有用户可见变化，同步更新文档和变更记录

## 建议输出结构
```markdown
### 问题
- 现象
- 影响

### 根因
- 直接原因
- 深层原因

### 修复
- 改了哪些文件
- 为什么这样改
- 是否保持兼容

### 验证
- 运行了哪些测试
- 结果如何

### 风险
- 仍需关注的边界
- 回滚方式
```

## 约束
- 不要用大改动掩盖问题
- 不要在没复现前直接下结论
- 不要跳过测试和验证
- 如果需要文档同步，修复后要一并补上
