# analyze-pr

## 目标
对单个 PR 做结构化审查，重点确认是否值得合入、是否存在回归风险，以及验证是否充分。

## 适用场景
- 用户要求 review / code review / PR 审查
- 需要给出“能不能合”“哪里有风险”“还缺什么验证”
- 需要把 PR 描述和实际改动对齐检查

## 审查原则
- 先看必要性，再看正确性
- 先看高风险改动，再看普通改动
- 以证据为主，不以直觉拍板
- 与 `AGENTS.md`、CI 结果、仓库测试约定保持一致

## 推荐工作流
1. 读 PR 描述、变更范围和相关 issue
2. 对照仓库规范判断改动是否必要、是否越界
3. 检查实现正确性：边界条件、兼容性、失败路径、回退逻辑
4. 检查验证证据：单测、集成测试、CI、日志或手工验证
5. 判断风险：数据链路、通知链路、配置语义、回滚复杂度
6. 给出合入建议：通过、修改后通过、拒绝

## 建议输出结构
```markdown
### 结论
- 能否合入
- 核心理由

### 必要性
- 是否解决了明确问题
- 是否和描述一致

### 实现正确性
- 正确点
- 风险点
- 兼容性问题

### 验证情况
- 已验证内容
- 未覆盖内容

### 合入建议
- 直接合入 / 修改后合入 / 拒绝
```

## 约束
- 不要只看代码表面，不看 CI 和测试
- 不要把风格偏好当成阻断理由
- 不要忽略回滚和兼容风险
- 如果 PR 和描述不一致，要明确指出
