# analyze-issue

## 目标
对单个 issue 做结构化分析，帮助后续修复或评审快速定位问题，不直接改代码。

## 适用场景
- 用户要求“看看这个 issue 怎么回事”
- 需要把用户反馈整理成可执行的修复建议
- 需要确认问题是代码缺陷、配置问题、数据问题还是预期变更

## 分析原则
- 先给结论，再给证据
- 先判断影响面，再判断修复方式
- 只给可验证的判断，不做无依据推测
- 与 `AGENTS.md` 保持一致，不引入仓库外流程

## 推荐工作流
1. 读 issue 原文、相关日志、截图或复现步骤
2. 定位问题类型：功能缺陷 / 回归 / 配置 / 数据 / 文档 / 预期变更
3. 判断影响范围：单点、模块级、链路级、全局
4. 追踪根因：优先看最近改动、回归测试、错误日志、运行态证据
5. 给出处理建议：最小修复、是否需要补测试、是否需要补文档
6. 输出验证点：修复后如何确认问题已经消失

## 建议输出结构
```markdown
### 结论
- 一句话说明问题本质

### 影响范围
- 受影响模块
- 影响用户行为
- 是否阻断主流程

### 根因判断
- 直接原因
- 深层原因
- 证据

### 修复建议
- 最小修复方案
- 是否需要测试补强
- 是否需要文档同步

### 验证点
- 复现步骤
- 预期结果
```

## 约束
- 不要把猜测写成事实
- 不要忽略回归风险
- 不要直接给大重构建议替代问题定位
- 如果证据不足，要明确标注“不足以确认”
