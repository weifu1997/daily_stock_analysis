## 结构性结论
- 项目里存在“策略表达层”：`src/core/market_strategy.py` 的 `MarketStrategyBlueprint` / `StrategyDimension`，用于市场复盘蓝图与分析维度描述。
- 项目里存在“策略配置层”：`tests/test_agent_registry.py` 验证了 YAML 策略/技能加载能力，说明策略定义可配置化。
- 项目里已经形成完整的“策略骨架”链路：`Skill`（`src/agent/skills/base.py`）负责策略定义，`SkillRouter`（`src/agent/skills/router.py`）负责策略选择，`pipeline`（`src/core/pipeline.py`）负责执行，`.env` 则提供 `AGENT_SKILLS` / `AGENT_SKILL_ROUTING` 等控制开关。
- 项目里有完整的内置策略文件：`strategies/*.yaml` 下包含 `bull_trend`、`dragon_head`、`shrink_pullback`、`volume_breakout`、`ma_golden_cross`、`box_oscillation` 等策略，说明策略文本已经文件化，不只是代码逻辑。
- `src/agent/skills/defaults.py` 统一管理默认技能基线、默认激活策略和按市场状态映射的策略集合；`src/agent/factory.py` 负责读取 `AGENT_SKILLS` / `AGENT_SKILL_ROUTING`、生成 `SkillPromptState` 并激活策略。
- 当前策略体系的真实形态是“技能化策略编排系统”：策略定义在 YAML，默认规则在 `defaults.py`，路由在 `router.py`，装配在 `factory.py`，执行在 `pipeline.py`。
- `defaults.py` 已显式算出 `DEFAULT_ACTIVE_SKILL_IDS`、`DEFAULT_ROUTER_SKILL_IDS`、`PRIMARY_DEFAULT_SKILL_ID` 和 `REGIME_SKILL_IDS`，说明默认激活、默认路由、主默认策略以及市场状态到策略的映射都在启动层形成。
- market regime 绑定已经明确：例如 `bull_trend -> trending_up`，`shrink_pullback -> trending_down/sideways`，`dragon_head -> sector_hot`，`box_oscillation -> sideways`，路由是按市场状态分发战法而非临时拍脑袋选择。
- 目前尚未确认存在传统量化框架那种独立 `StrategyBase`、信号引擎、参数中心、回测闭环的完全分层实现。

## 妙想技能接入判断
- 当前项目已明确使用 `mx-data` 和 `mx-search`；其中 `MxClient.search()` 在带 `code/name/stock_code/stock_name` 参数时，会转到 `stock-screen`，这实际上已经利用到了 `mx-xuangu` 的入口能力。
- `.env` 里已经存在妙想预选池风格入口：`MX_PRESELECT_PROFILE=fundamental`，并预留 `trend` / `fundamental` / `basic` 风格，以及 `MX_PRESELECT_PRIORITY`、`MX_PRESELECT_QUERY`、`MX_PRESELECT_LIMIT`，说明配置层早已预置预选池风格控制。
- 现阶段代码里已确认有 `stock-screen` 入口、`CandidateEnrichmentService`、`MxSearchAdapter` 等能力；`StockAnalysisPipeline.run()` 现在已进入第三步改造，`_build_candidate_pool()` 会先保留原始输入，再并入持仓池，并在 `mx_preselect_priority=True` 时调用 `CandidateEnrichmentService.enrich_candidates()` 做妙想预选。
- 当前 `mx-xuangu` 的接入逻辑是：原始输入 + 持仓池组成预筛种子池 → 通过 `mx_event_score` 排序 → 按 `mx_preselect_limit` 截断 → 持仓股再强制并回最终池 → 生成 `final_candidate_pool`。
- `mx-xuangu` 最合理的接法是接在候选池生成层，而不是直接替换本地选股战法：它应该先缩小全市场范围，再把候选池交给本地 skill 战法做二次筛选。
- `mx-xuangu` 最安全的插入点是在 `StockAnalysisPipeline.run()` 中、`prefetch_realtime_quotes()` 之前、线程池提交之前；此时还没开始并发分析，最容易调整候选池，且不会污染单股分析内部逻辑。
- `mx-xuangu` 接入时建议返回双层结构：保留 `original_stock_codes`，另存 `candidate_pool`，并记录 `mx_xuangu_enabled`、`mx_xuangu_reason`、`mx_xuangu_source`，以便随时回退。
- 为最大程度消除副作用，建议：永远不覆盖原始输入、提供明确 fallback（原始 `stock_codes` 或 `STOCK_LIST`）、限制筛选强度（保留最小数量）、把妙想结果定位成“建议”而不是“裁决”，并加入分层日志（命中/为空/异常/fallback）。
- `mx-xuangu` 的正面影响：候选池更准、主流程更省、策略命中率更容易提高；潜在副作用：候选池受妙想规则偏好影响、外部服务稳定性影响预筛结果、与本地战法可能存在取票偏差。
- `mx-zixuan` 最合理插入点是自选池 / 观察池 / 持仓闭环层：对应 `portfolio_service`、`portfolio_repo`、`get_portfolio_snapshot`、`PortfolioAgent`，用于把精选结果沉淀为长期跟踪对象。
- `mx-moni` 最合理插入点是回测 / 模拟交易验证层：对应 `backtest_engine`、`backtest_service`、`backtest_tools`，用于把策略建议转成模拟执行并验证行为闭环。
- 三者优先级建议：`mx-xuangu` > `mx-zixuan` > `mx-moni`。
- 结合代码实际，最合理的接法是：`mx-xuangu` 先选股，`mx-search` / `mx-data` 再增强，本地策略技能再判战法，`mx-zixuan` 结果落自选池，`mx-moni` 做模拟验证。
- `mx-xuangu` 的具体接入进度：`StockAnalysisPipeline.run()` 已新增 `_build_candidate_pool()` 骨架，并在正式分析前统一读取 `candidate_bundle`；当前骨架已能保留原始输入、并入持仓池，并在 `mx_preselect_priority=True` 时进行妙想预选排序与限额截断，但仍建议后续做运行验证与阈值调优。
- 持仓池股票必须强制并入分析池，最终候选池应是“原始输入 + 持仓池 + 妙想预选”的并集，且要去重，避免持仓股漏分析。
